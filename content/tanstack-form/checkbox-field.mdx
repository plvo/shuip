## Standalone checkbox integration

The simple checkbox field pattern provides a straightforward way to implement boolean toggles with TanStack Form. Each field receives the form instance as a prop, offering excellent type inference and explicit data flow.

## Core characteristics

- **Zero configuration**: No context setup required
- **Type-safe field names**: Autocomplete for nested paths via `DeepKeys<TFormData>`
- **Boolean values**: Handles `true`/`false` states
- **Explicit dependencies**: Form instance passed directly to each field
- **Dual labeling**: Supports both field label and checkbox-specific label

## Basic usage

<CodeHighlight code={`import { useForm } from '@tanstack/react-form'
import { CheckboxField } from '@/components/ui/tanstack-form/checkbox-field'

function TermsForm() {
  const form = useForm({
    defaultValues: {
      terms: false,
      newsletter: false,
    },
    onSubmit: async ({ value }) => {
      if (value.terms) {
        await createAccount(value)
      }
    },
  })

  return (
    <form
      onSubmit={(e) => {
        e.preventDefault()
        form.handleSubmit()
      }}
    >
      <CheckboxField
        form={form}
        name='terms'
        label='Terms and Conditions'
        boxLabel='I accept the terms and conditions'
        description='You must accept to continue'
      />

      <CheckboxField
        form={form}
        name='newsletter'
        label='Newsletter'
        boxLabel='Subscribe to our newsletter'
        description='Optional: Receive updates about new features'
      />

      <button type='submit'>Sign Up</button>
    </form>
  )
}`} language='tsx' />

## Type inference

The component provides full type safety for boolean values:

<CodeHighlight code={`type FormData = {
  terms: boolean
  privacy: boolean
  marketing: boolean
}

const form = useForm<FormData>({
  defaultValues: {
    terms: false,
    privacy: false,
    marketing: false,
  }
})

// TypeScript ensures 'terms' is a valid boolean field
<CheckboxField
  form={form}
  name='terms'
  label='Agreements'
  boxLabel='I agree to the terms'
/>`} language='tsx' />

The `name` prop is constrained to `DeepKeys<TFormData>`, ensuring only valid field paths are accepted.

## Validation integration

TanStack Form supports validation on checkbox fields:

<CodeHighlight code={`const form = useForm({
  defaultValues: { terms: false },
})

<form.Field
  name='terms'
  validators={{
    // Require checkbox to be checked
    onChange: ({ value }) =>
      !value ? 'You must accept the terms and conditions' : undefined,

    // Custom validation
    onBlur: ({ value }) =>
      value && !hasReadTerms()
        ? 'Please read the terms before accepting'
        : undefined,
  }}
>
  {(field) => (
    <CheckboxField
      form={form}
      name='terms'
      label='Terms'
      boxLabel='I have read and accept the terms'
    />
  )}
</form.Field>`} language='tsx' />

## Required checkbox

Implement a required checkbox pattern:

<CodeHighlight code={`<form.Field
  name='terms'
  validators={{
    onChange: ({ value }) =>
      !value ? 'Required' : undefined,
  }}
>
  {(field) => (
    <CheckboxField
      form={form}
      name='terms'
      label='Terms and Conditions'
      boxLabel='I accept the terms and conditions (required)'
      required
    />
  )}
</form.Field>`} language='tsx' />

## Conditional visibility

Show/hide checkboxes based on other field values:

<CodeHighlight code={`function NotificationForm() {
  const enableNotifications = useWatch({

        form={form}
        name='enableNotifications'

## Accessibility

The component implements proper accessibility patterns:

- Checkbox has a unique `id` based on field name
- Label is properly associated via `htmlFor`
- ARIA attributes for invalid state
- Keyboard navigation (Space to toggle)
- Focus ring for keyboard users
- Screen reader support

## When to use this pattern

**Ideal for:**
- Simple forms with fewer than 10 fields
- Quick prototypes
- Boolean toggles and agreements
- Forms without complex composition needs
- Terms acceptance, preferences, opt-ins

**Consider the composed pattern when:**
- Building multiple forms across your app
- Creating reusable form component libraries
- Need lazy loading for large forms
- Want to minimize props on field components
